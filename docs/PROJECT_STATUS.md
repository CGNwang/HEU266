# 项目现状与核心缺口（2026-04-19）

本文档是当前唯一的项目状态基线，用于替代历史上已过时的实现总结与分阶段指南。

## 1. 当前已落地能力

### 1.1 账号与访问控制
- 邮箱注册、登录、登出、重置密码、注销账号流程已接入。
- 支持 Supabase 与本地降级模式双路径运行。
- 问卷、等待、聊天、个人中心等核心内页已统一登录守卫，未登录将重定向到登录页。

### 1.2 问卷系统
- 五模块问卷页面与前端进度逻辑已落地。
- 问卷答案可本地保存，并支持与 Supabase 的答案表同步读取/写入。
- 已有问卷完成态判断，支持页面分流。

### 1.3 聊天系统（基本完成，需要测试完善）
- 基础消息收发、实时订阅、解盲申请、举报、拉黑、联系方式交换入口已具备。
- 无有效匹配时存在演示兜底模式，保障页面可访问。

### 1.4 其他功能
- 反馈工单提交已接入。
- 通知偏好设置页与持久化已接入。
- 等待页支持按现实时间倒计时到下次揭晓时刻。

## 2. 核心未完成项（按优先级）

### P0：真实匹配链路仍未闭环
- matchingService 主要接口仍以本地模拟为主，尚未完全切换到真实后端匹配结果。
- 等待页仍保留测试跳转入口，发布前需移除。

### P0：匹配报告仍为模拟数据
- 匹配报告页目前使用前端固定示例数据。
- 需接入 match_reports 真实数据，并与 matchId 关联加载。

### P0：匹配算法与调度仍是占位实现
- 文档内 edge function 的问卷相似度计算仍是占位分。
- 需要可解释的真实评分函数、阈值策略、失败原因记录与回溯。

### P1：绑定页业务未接后端（微信绑定都需要组织资质，才能有完整功能，暂时不做；手机绑定似乎也有资质限制）
- 绑定页当前仍是 UI 跳转逻辑，未接微信/手机绑定能力。

### P1：聊天时效与结束匹配未闭环
- 聊天倒计时需基于真实匹配的 expires_at。
- 结束匹配按钮未接业务操作。

### P1：通知发送能力未落地
- 目前仅有通知偏好设置，缺少真正的发送任务（站内/邮件/短信）。

### P2：数据库初始化步骤分散（已完成）
- 部分功能依赖增量 SQL，不执行会导致功能页面可见但接口失败。

## 3. Supabase 初始化最小执行顺序（已完成）

按以下顺序执行 SQL，避免表与策略缺失：

1. docs/database-schema.sql
2. docs/database-rls.sql
3. docs/auth-email-domain-constraint.sql
4. docs/delete-account-function.sql
5. docs/semi-anonymous-chat-schema.sql
6. docs/feedback-tickets-schema.sql

说明：前四项是基础能力，后两项是匿名聊天与反馈工单的增量能力。

## 4. 发布前必做检查清单

### 4.1 功能正确性
- 未登录访问受限页面应全部重定向登录页。
- 问卷完成、等待页、聊天入口分流逻辑一致。
- 匹配成功/失败/报告页全部来自真实后端数据。

### 4.2 安全与数据
- RLS 对 profiles、questionnaire_answers、matches、chat_messages 以及增量表全部生效。
- 举报与拉黑后，消息写入策略按预期生效。

### 4.3 体验收口
- 移除所有测试入口与演示文案。
- 替换演示兜底数据为真实空态提示。
